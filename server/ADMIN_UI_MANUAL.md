# 管理UIマニュアル（管理者ID＋パスワード版）

このドキュメントは、管理画面（/admin/ui）のログイン方法と基本操作をまとめたものです。2026年3月以降、管理者IDとパスワードによるサインインが標準となりました。

## 0. 新規セットアップ（初期ブートストラップ）

新規環境では、管理UIの「初期セットアップ（新規向け）」から次を一括生成できます。

- 管理トークン（ADMIN_TOKEN, 互換用）
- 初期テナントID
- 最初の管理者ID／パスワード

手順:

1. `https://<サーバIP>/admin/ui/` を開く
2. 必要ならテナント名、必須の管理者ID・パスワード・表示名を入力
3. 「管理トークン＋テナントIDを自動生成」を押す
4. レスポンスに含まれる `tenant_id` と指定した管理者IDでログイン

注意:

- 初期化APIは「テナント0件 かつ ADMIN_TOKEN未設定」の状態でのみ実行可能です。
- ADMIN_TOKEN は互換用として `.env` に書き込まれますが、通常操作は管理者ログインで行ってください。

## 1. ログインとセッション

画面上部「管理者ログイン」から ID / パスワードを入力します。成功すると以下が行われます。

- ブラウザの localStorage に短期セッションが保存され、以降の API 呼び出しへ自動付与
- 画面右上に「ログイン中: <表示名>」と表示
- 「ログアウト」ボタンでセッションとブラウザ保存内容をクリア

セッションが失効するとステータスが赤表示になり、再ログインが必要です（デフォルト有効期限: 12時間）。

## 2. 管理者アカウントの追加/一覧

「管理者アカウント」セクションで追加・閲覧が可能です。

- *新規管理者ID*: 3〜64文字。
- *初期パスワード*: 12文字以上を推奨（サーバ側でも下限を enforcing）。
- 追加後は `管理者一覧を再読み込み` で即時反映。
- 一覧には ID / 表示名 / DB上のID が表示され、どの管理者がアプリを登録したかも UI や API レスポンスに表示されます。

## 3. レガシー ADMIN_TOKEN 互換

従来の ADMIN_TOKEN 方式も fallback として利用できます。

1. `.env` の `ADMIN_TOKEN` を確認（または再生成）
2. 管理UIの「レガシートークン保存（旧互換機能）」に入力して保存
3. セッションが無い場合でも `X-Admin-Token` ヘッダーで管理APIを実行

> **推奨:** 日常運用は ID/パスワード＋セッションで行い、ADMIN_TOKEN は緊急用に限定してください。

## 4. 主な UI 操作

### 4-1. テナント作成

- 「テナント作成」で名称を入力、`tenant_id` が払い出されます。
- 直近のテナントIDは自動的に他フォームへ補完されます。

### 4-2. アプリ登録

- テナントID・Kintoneドメイン・アプリID・認証方式・APIトークンを入力。
- レスポンスには `app_id` と、可能であれば自動取得したフォーム/ビュー定義が含まれます。
- 2026年3月以降、登録結果には「登録管理者」が付与され、一覧画面・APIの `created_by_admin` フィールドで確認できます。

### 4-3. スキーマ登録

- `app_id` と JSON をアップロードして form/view 定義を追加更新。

### 4-4. アクセストークン発行

- scope, 有効期限（分）, 任意の record_ref を指定すると一時的なアクセストークンを払い出します。

### 4-5. アプリ一覧（テナント別）

- テナントIDを指定して読み込み。各行に ID / ドメイン / アプリコード / 認証種別 / マスク済トークン / 登録管理者 / 画面リンク が表示されます。
- 画面リンクから次を直接開けます（別タブ）。
	- フォーム: `/<tenant>/<app>/form`
	- 一覧: `/<tenant>/<app>/viewer/ui`
	- レコード操作: `/<tenant>/<app>/record/ui`

### 4-5-1. レコード一覧（viewer）

- `/<tenant>/<app>/viewer/ui` では、実際のKintoneレコード一覧を表示できます。
- 一覧にはレコードID・revision・内容プレビューが表示され、各行の「編集」からレコード編集画面へ遷移できます。
- 画面上部の query 入力欄で Kintone query 条件を指定して絞り込み取得できます。

### 4-5-2. レコード表示・編集（record）

- `/<tenant>/<app>/record/ui` では、レコードID指定で実データを取得し、編集して更新できます。
- 既存互換としてトークン指定（view/edit）での参照・更新も継続利用できます。
- 更新時は JSON オブジェクト形式で入力してください（`$id`, `$revision` 等のシステム項目は通常編集対象外）。

### 4-6. アプリ一覧（全テナント横断）

- 「全件読み込み」で全テナントの登録済アプリを確認できます。列構成: tenant_id, tenant_name, app_id, domain, app_code, auth_type, token(masked), 登録管理者, 画面リンク。
- 各行のリンクはテナントIDとアプリIDを使って自動生成されます。
- 各行の「テナント削除」から、対象テナントのローカル管理データ（tenant/app/schema/token）を削除できます。
- この削除は **Kintone側のアプリ・レコードには影響しません**（連携サーバ側の管理データのみ削除）。

### 4-7. 登録済みテナント一覧とフォームリンク

- 「登録済みテナント一覧」セクションで「一覧取得」を押すと、テナントごとの登録アプリをカード形式で確認できます。
- 各アプリ行から「フォーム / 一覧 / レコード」の各画面へ遷移できます。
- 同じ画面の「操作」列からテナント削除も可能です（ローカルのみ）。
- フォーム画面（`/<tenant>/<app>/form`）は自動取得したKintoneスキーマをもとに入力フィールドを動的生成し、申請・ユーザ入力用の新規レコード登録画面として利用できます。

## 5. 代表的なエラー

- `401 unauthorized`: 未ログイン／セッション切れ／トークン不一致。ログインし直すかレガシートークンを更新してください。
- `admin_auth_not_configured`: 管理者アカウントも ADMIN_TOKEN も設定されていない状態。初期化手順を完了させる必要があります。
- `already_initialized`: 既存環境で再度ブートストラップ API を呼び出した場合。
- `400 invalid`: 必須フィールド不足、JSONパース失敗などクライアント側の入力エラー。

## 6. セキュリティ運用メモ

- 管理者パスワードは 12 文字以上を推奨し、共有せず個人ごとに発行してください。
- ログアウト後は共有端末のブラウザ localStorage をクリアしてください。
- ADMIN_TOKEN を更新した場合は `sudo systemctl restart kintone-bridge` でサーバを再起動してください。
- `.env` は Git 管理に含めず、権限を限定した状態で保管します。

以上で管理者IDベースの運用へ移行できます。既存のスクリプトや監視が ADMIN_TOKEN に依存している場合は、段階的にセッション方式へ移行してください。
